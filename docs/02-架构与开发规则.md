# 架构与开发规则

## 先用白话说清楚

以后这个项目不要再按“哪里报错就改哪里”的方式发展。

要固定成下面这套分工：

- `main.go`：前台接待，负责 HTTP、WebSocket、请求调度
- `scanner/`：真正执行网络探测
- `static/`：页面展示和交互
- `scripts/check.ps1`：统一验收入口
- `*_test.go`：自动回归保护

一句话：

**UI 负责展示，调度负责串流程，扫描负责干活，测试负责判分。**

## 目录职责

### `main.go` / `request_logic.go`

这里放：

1. 服务入口
2. WebSocket 请求处理
3. 轻量调度逻辑
4. 适合抽成纯函数、方便测试的请求处理规则

这里不要放：

1. 大量前端展示逻辑
2. 具体测速实现细节
3. 一堆临时 if/else 拼起来的业务规则

### `scanner/`

这里放：

1. `Ping()`
2. `SpeedTest()`
3. 未来新增的底层扫描能力

这里不要放：

1. 页面状态
2. WebSocket 写回逻辑
3. 前端输入格式处理

### `static/`

这里放：

1. 页面元素
2. 表单交互
3. 结果展示
4. 前端排序和状态提示

这里不要放：

1. 真正的扫描规则
2. 和后端重复的核心业务判断

## 开发规则

### 规则 1：先写“什么算通过”

每次改动前，先写清楚：

1. 本次目标是什么
2. 哪个行为算通过
3. 用什么命令检查

不能只说“帮我优化一下”。

### 规则 2：先补测试，再改实现

优先顺序固定：

1. 先补或更新测试
2. 再改代码
3. 再跑 `scripts/check.ps1`
4. 失败就继续修，直到通过

### 规则 3：优先抽纯逻辑

如果一段规则能不碰网络、不碰 WebSocket、不碰浏览器，就先抽成普通函数。

原因：

- 纯逻辑最容易测
- 纯逻辑最适合 AI 安全修改

### 规则 4：不要一上来重写大块代码

每次只改一个明确目标。

不要出现这种动作：

1. 修一个 bug，顺手把整个模块重写
2. 还没补测试，就先大改结构
3. 前端后端一起大面积改动，最后说不清是哪层出了问题

## 标准开发循环

以后默认按这个顺序干活：

```text
1. 写清本次目标
2. 写清验收标准
3. 补测试或补检查
4. 修改代码
5. 运行 scripts/check.ps1
6. 看失败信息
7. 继续修到通过
```

## AI 改代码时的固定输入模板

以后给 AI 下任务，尽量带上这 6 行：

```text
本次目标：
本次只允许改哪些文件：
哪些地方不要动：
验收标准：
检查命令：
失败后先看什么输出：
```

例子：

```text
本次目标：修复 CIDR 输入展开错误
本次只允许改哪些文件：main.go、request_logic.go、main_test.go
哪些地方不要动：static/ 前端页面
验收标准：CIDR 展开正确，原有测试全部通过
检查命令：.\scripts\check.ps1
失败后先看什么输出：go test 失败信息
```

## 当前阶段最重要的纪律

1. 不要把“手工点页面看看”当成唯一测试方式。
2. 不要把核心规则塞进 `static/app.js`。
3. 不要把网络波动问题误判成代码逻辑问题。
4. 不要在没有测试保护的情况下做大改。

## 一句话总结

这个项目以后要从“凭感觉改”切到“先定义通过标准，再按规则开发，最后用统一命令自动判分”。
