# 项目目标与验收

## 这项目是干什么的

`CFping` 是一个带网页界面的 Cloudflare IP 扫描工具。

它当前要解决的核心问题只有两件事：

1. 用户输入 IP 或 CIDR 后，系统能发起 `ping` 或测速任务。
2. 前端能实时看到结果，而不是等全部任务结束再一起显示。

你可以把它理解成一个小型检测台：

- 前端页面负责“下单”
- Go 后端负责“派工”
- `scanner` 负责“真正干活”

## 当前版本范围

这一版先保证下面这些能力稳定可用：

1. 服务器能在本地启动，网页能打开。
2. WebSocket 能建立连接并接收扫描结果。
3. 后端能处理单个 IP 和 CIDR 输入。
4. `ping` 任务能返回延迟或错误状态。
5. `speed` 任务能返回下载速度或错误状态。
6. 超过最大目标数时，后端会截断请求，避免一次性扫太多地址。

## 暂时不承诺的东西

下面这些不算当前阶段必须解决的问题：

1. 外网测速结果绝对稳定。
2. 不同地区、不同网络环境下结果完全一致。
3. 一次扫描超大网段。
4. 复杂的历史记录、任务管理、取消任务、导出结果。

原因很简单：

- 外网测速会受到真实网络波动影响
- 这些波动不适合拿来做第一层自动测试

所以第一层自动化要先锁住“本地可稳定判分的行为”，再去碰真实网络。

## 第一版验收标准

### A. 启动与基础连接

1. `go build ./...` 成功。
2. `go test ./...` 成功。
3. 启动后可以访问 `http://localhost:13334`。
4. 页面打开后 WebSocket 能连上 `/ws`。

### B. 后端目标处理

1. 普通 IP 输入能被正确保留。
2. CIDR 输入能被正确展开。
3. 非法 CIDR 不会让程序崩溃，只会跳过并记录日志。
4. 目标总数超过 `10000` 时会被截断。

### C. Ping 行为

1. 对可连接端口，`Ping()` 返回 `ok`。
2. 对不可连接端口，`Ping()` 返回 `error` 且带错误信息。
3. 当 `ping_time` 超过 `max_latency` 时，该条结果会被过滤掉。

### D. Speed 行为

1. 对可访问下载地址，`SpeedTest()` 返回 `ok`。
2. 对非法下载地址，`SpeedTest()` 返回 `error` 且带错误信息。

### E. 前端基本行为

1. 点击按钮后能发送正确请求。
2. 扫描完成后按钮恢复可点击状态。
3. 结果表格能正常追加和排序。

## 完成定义

以后这个项目里“本次改动完成”至少要满足下面 4 条：

1. 本次目标写清楚了。
2. 对应测试或检查已经补上或更新。
3. `scripts/check.ps1` 通过。
4. 没有顺手改出和本次目标无关的大范围变化。
